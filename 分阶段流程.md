# 分阶段流程

## 1. 接收任务

FastAPI 接收 VIN、用户主诉、DTC、冻结帧、车辆信息和参数，创建诊断任务。

## 2. 执行前置 Agent

```text
历史故障和维修记录 Agent
  -> 用户主观描述 Agent
  -> DTC Agent
  -> RAG 知识库 Agent
  -> 汇总/辩论 Agent
```

说明：

- `vin_context` 和 `symptom` 在职责上相互独立，工程上可以并行优化；当前 LangGraph 实现保持顺序执行以便状态合并和进度追踪更稳定。
- `symptom` 只负责语义标准化、疑似故障点和排查方向。
- `dtc` 只负责按方向查询 DTC/冻结帧/数据流事实。
- `knowledge` 只负责查车辆手册、技术资料和历史案例。
- 所有交叉验证、冲突仲裁和最终判断都放在 `Summary Agent`。

## 3. 工具调用

保留工具调用：

- VIN 车辆档案、历史故障码、维修保养记录
- DTC 字典和故障码组合查询
- 知识库/案例库查询
- outcome 确认后的经验库写入

数据库查询通过 Skill/Tool 封装，LLM 只传参数，不直接生成 SQL。

## 4. 汇总输出

`Summary Agent` 根据前序报告生成：

- 最可能故障排序
- 支持证据和反证
- 主观、客观、RAG 资料之间的辩论说明
- 下一步检查建议
- 置信度
- 完整 trace 和工具错误

## 5. 经验写入

维修完成并确认根因后，通过 outcome API 写入经验库。系统不会把未确认的推测自动写入经验库。
