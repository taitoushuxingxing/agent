# 车辆故障诊断 Agent 工作流程说明

本文档用于说明 `vehicle_fault_diagnosis_agent` 项目的整体运行流程，重点覆盖：

- API 请求进入系统后的任务流转
- LangGraph 多 Agent 诊断链路
- 分析师节点如何调用工具
- 记忆 Memory 的读取、写入和持久化方式
- 工具、LLM、数据源、任务执行中的 fallback 与容错策略
- 最终诊断结果如何生成、存储和查询

项目整体定位是一个面向车辆故障诊断场景的后端 Agent 系统。它使用 FastAPI 提供服务接口，Redis 承载异步任务队列和请求限流，MongoDB/SQLite 承载任务、结果、诊断状态和反馈记忆，LangGraph 编排多角色 Agent 完成“数据采集、证据分析、假设辩论、诊断规划、维修建议、安全评审”的诊断闭环。

## 1. 系统分层

项目可以按职责拆成五层：

| 层级 | 主要目录/文件 | 职责 |
| --- | --- | --- |
| API 层 | `app/main.py`, `app/routers/vehicle_diagnosis.py` | 提供 HTTP 接口，接收诊断请求、查询状态/结果、取消任务、回填维修结果 |
| 服务编排层 | `app/services/vehicle_diagnosis_service.py` | 创建任务、入队、worker 执行、进度保存、超时/取消处理、Graph 缓存 |
| Agent 图编排层 | `vehicleagents/graph/*.py` | 构建 LangGraph，定义节点顺序、条件路由和最终结果处理 |
| Agent 节点层 | `vehicleagents/agents/**` | 各类分析师、研究员、规划器、维修建议、安全评审节点 |
| 数据与工具层 | `vehicleagents/dataflows/**`, `vehicleagents/agents/utils/agent_utils.py` | 对 VIN、DTC、传感器、事件、维修案例等数据源封装成可调用工具 |

运行时的主链路是：

```text
HTTP 请求
  -> FastAPI Router
  -> VehicleDiagnosisService
  -> Repository 创建任务
  -> Redis Queue 入队
  -> Worker 取任务
  -> VehicleDiagnosisGraph.diagnose()
  -> LangGraph 多节点执行
  -> structured_result
  -> Repository 保存结果
  -> API 查询状态/结果
```

## 2. API 层工作流程

API 路由集中在 `app/routers/vehicle_diagnosis.py`。

### 2.1 提交诊断任务

接口：

```http
POST /api/vehicle-diagnosis/tasks
```

请求体由 `VehicleDiagnosisRequest` 校验，核心字段包括：

| 字段 | 说明 |
| --- | --- |
| `vin` | 车辆 VIN，可直接传，也可从 `vehicle.vin` 中补齐 |
| `vehicle` | 用户已知的车辆信息，如品牌、车型、里程、动力类型等 |
| `symptoms` | 用户描述的症状列表，例如怠速抖动、加速无力、异响 |
| `dtc_codes` | 用户或诊断仪读到的故障码，例如 `P0301`, `P0171` |
| `sensor_snapshot` | 可选的传感器快照 |
| `freeze_frame` | 可选冻结帧数据 |
| `maintenance_history` | 可选维修保养历史 |
| `user_question` | 用户原始问题 |
| `parameters` | 控制 Agent 行为的参数 |

`parameters` 支持：

| 参数 | 说明 |
| --- | --- |
| `selected_analysts` | 选择启用哪些分析师，支持 `vin_context`, `symptom`, `dtc`, `telemetry`, `knowledge` |
| `diagnosis_depth` | 诊断深度，支持 `quick`, `basic`, `standard`, `deep`, `comprehensive` |
| `max_debate_rounds` | 正反研究员辩论轮数上限 |
| `max_safety_discuss_rounds` | 维修建议和安全评审讨论轮数上限 |
| `max_tool_calls` | 全局单分析师最大工具调用次数 |
| `analyst_max_tool_calls` | 按分析师设置最大工具调用次数 |
| `tool_timeout_seconds` | 单次工具调用超时时间 |
| `tool_max_retries` | 工具调用失败后的重试次数 |

提交任务时不会同步跑完整诊断，而是：

1. Router 调用 `VehicleDiagnosisService.submit_task()`。
2. Service 先创建 `DiagnosisTask`，初始状态为 `submitted`。
3. Repository 将任务持久化。
4. Service 调用 `enqueue_task()`，把任务状态改为 `queued`，写入队列位置、进度 `5`。
5. Task ID 被放入 Redis 队列。
6. API 返回 `task_id` 和 `queued` 状态。

### 2.2 查询任务状态

接口：

```http
GET /api/vehicle-diagnosis/tasks/{task_id}/status
```

状态返回由 `DiagnosisTask.to_status()` 生成，包含：

- `status`: 当前任务状态
- `progress`: 进度百分比
- `current_step`: 服务层当前步骤
- `current_node`: LangGraph 当前节点
- `graph_trace`: 节点执行轨迹
- `analyst_conclusions`: 各分析师的简短结论
- `tool_errors`: 工具调用中可恢复的错误
- `queue_position`: 排队位置
- `error_message`: 失败原因
- 创建、更新时间、开始时间、完成时间

### 2.3 查询诊断结果

接口：

```http
GET /api/vehicle-diagnosis/tasks/{task_id}/result
```

如果任务未完成，返回当前状态和错误信息；如果完成，返回 `structured_result`，即 Safety Judge 生成的最终结构化诊断结果。

### 2.4 取消任务

接口：

```http
POST /api/vehicle-diagnosis/tasks/{task_id}/cancel
```

取消逻辑分两种：

| 当前状态 | 处理方式 |
| --- | --- |
| `submitted` / `queued` | 尝试从队列移除，直接标记为 `cancelled` |
| `running` | 标记为 `cancel_requested`，Graph 执行流在进度回调里检查到后抛出取消异常，最终落为 `cancelled` |

当前实现不会强杀正在运行的 worker 线程，而是在 Graph streaming 的进度回调中协作式检查取消信号。

### 2.5 回填维修结果

接口：

```http
POST /api/vehicle-diagnosis/tasks/{task_id}/outcome
```

该接口用于真实维修完成后，把确认根因和维修动作写回系统记忆。请求体包括：

| 字段 | 说明 |
| --- | --- |
| `confirmed_root_cause` | 真实确认的故障根因 |
| `repairs_performed` | 实际执行的维修动作 |
| `resolved` | 是否解决问题 |
| `notes` | 可选备注 |

Service 会将当前任务请求、症状、故障码、遥测发现和维修结果整理成 memory case，写入 `VehicleDiagnosisMemory`。

## 3. 服务层任务执行流程

服务层核心类是 `VehicleDiagnosisService`。

### 3.1 初始化

服务启动时，FastAPI lifespan 会执行：

```text
service.initialize()
service.start_workers()
```

初始化内容包括：

1. 初始化任务 Repository。
2. 初始化 Redis 队列。
3. 加载 LLM 配置。
4. 创建 quick/deep 两类 LLM 客户端。
5. 根据配置启动多个 worker。
6. 将异常中断前处于 `submitted`, `queued`, `running` 的任务重新入队。

### 3.2 Repository 选择

默认生产配置使用：

```text
MongoVehicleDiagnosisRepository
```

它会写入 MongoDB 集合：

```text
vehicle_diagnosis_tasks
```

测试或注入场景可使用：

```text
VehicleDiagnosisRepository
```

该 Repository 使用 SQLite 表：

```text
vehicle_diagnosis_tasks
```

无论 MongoDB 还是 SQLite，保存的任务字段基本一致：

- `task_id`
- `request`
- `status`
- `progress`
- `current_step`
- `queue_position`
- `result`
- `state`
- `outcome`
- `error_message`
- `created_at`
- `updated_at`
- `started_at`
- `completed_at`

### 3.3 队列与 Worker

默认使用 Redis 队列：

```text
RedisTaskQueue
```

队列 key 来自：

```env
VEHICLE_DIAGNOSIS_REDIS_QUEUE=vehicle_diagnosis:tasks
```

核心行为：

- `put()` 使用 `RPUSH` 入队
- `get()` 使用 `BLPOP` 阻塞取任务
- `qsize()` 使用 `LLEN`
- `remove()` 使用 `LREM` 删除排队中的任务

测试环境可以注入 `InMemoryTaskQueue`，基于 `asyncio.Queue`，不依赖 Redis。

Worker 主循环：

```text
while True:
  task_id = queue.get()
  execute_task(task_id)
  queue.task_done()
```

worker 数量由：

```env
VEHICLE_DIAGNOSIS_WORKER_CONCURRENCY
```

控制。

### 3.4 单任务执行

`execute_task()` 的执行流程：

1. 读取任务。
2. 如果任务已完成，直接返回，避免重复执行。
3. 如果任务已取消，标记为 `cancelled`。
4. 将任务状态改为 `running`，进度改为 `10`。
5. 根据请求参数获取或创建 `VehicleDiagnosisGraph`。
6. 通过 `asyncio.to_thread()` 在线程中运行图诊断，避免阻塞事件循环。
7. 使用 `asyncio.wait_for()` 控制总超时。
8. Graph 每产生一个 snapshot，通过 progress callback 保存中间状态。
9. Graph 完成后保存最终 `state` 和 `result`。
10. 状态改为 `completed`，进度 `100`。

如果执行中发生异常：

| 异常类型 | 最终状态 | 说明 |
| --- | --- | --- |
| `asyncio.TimeoutError` | `failed` | 超过 `VEHICLE_DIAGNOSIS_TASK_TIMEOUT_SECONDS` |
| `TaskCancelledDuringGraph` | `cancelled` | 用户请求取消 |
| 其他异常 | `failed` | 保存异常文本到 `error_message` |

### 3.5 进度更新

Graph 支持 streaming 执行。每个 snapshot 会被 `_save_progress_snapshot()` 保存到任务 state。

进度估算逻辑 `_progress_from_state()` 参考：

- 选择的分析师数量
- 辩论轮数
- 安全讨论轮数
- `graph_trace` 长度
- 是否已经生成 `structured_result`

如果生成了 `structured_result`，进度接近完成，返回 `95`；真正保存最终结果后才变为 `100`。

### 3.6 Graph 缓存

Service 内部有 Graph 缓存：

```text
_graph_cache: OrderedDict[str, GraphCacheEntry]
```

缓存 key 由以下内容序列化而来：

- `selected_analysts`
- 诊断深度对应配置
- debate/safety 轮数
- 工具调用限制
- 工具超时/重试配置

缓存策略：

- 最大缓存数量：`VEHICLE_DIAGNOSIS_GRAPH_CACHE_MAX_SIZE`，默认 `8`
- TTL：`VEHICLE_DIAGNOSIS_GRAPH_CACHE_TTL_SECONDS`，默认 `1800`
- 超过数量时按 LRU 淘汰
- TTL 到期按 last used time 清理
- 设置 max size 为 `0` 时禁用缓存

这个设计避免每次任务都重新 compile LangGraph，降低重复请求的启动开销。

## 4. LangGraph 总体流程

Graph 入口类是：

```text
VehicleDiagnosisGraph
```

它负责：

1. 合并默认配置和请求配置。
2. 确定启用哪些分析师。
3. 创建 `VehicleConditionalLogic`。
4. 创建 `VehicleSignalProcessor`。
5. 调用 `VehicleGraphSetup.setup_graph()` 构建 LangGraph。
6. 在 `diagnose()` 中创建初始 state 并执行 graph。

默认启用的分析师顺序：

```text
vin_context -> symptom -> dtc -> telemetry -> knowledge
```

完整图流程：

```text
START
  -> VIN Context Analyst
  -> tools_vin_context?
  -> Msg Clear VIN Context
  -> Symptom Analyst
  -> tools_symptom?
  -> Msg Clear Symptom
  -> Diagnostic Code Analyst
  -> tools_dtc?
  -> Msg Clear Diagnostic Code
  -> Telemetry Analyst
  -> tools_telemetry?
  -> Msg Clear Telemetry
  -> Knowledge Analyst
  -> tools_knowledge?
  -> Msg Clear Knowledge
  -> Hypothesis Researcher
  -> Counterfactual Researcher
  -> ... debate loop ...
  -> Diagnostic Planner
  -> Repair Advisor
  -> Safety Analyst
  -> ... safety loop ...
  -> Safety Judge
  -> END
```

如果用户只选择部分分析师，例如：

```json
{
  "parameters": {
    "selected_analysts": ["dtc"]
  }
}
```

Graph 会只构建 DTC 分析师，然后直接进入研究、规划、维修建议和安全评审。

### 5.5 研究和评审状态

- `hypothesis_history`

### 5.6 最终输出

- `diagnostic_plan`

### 5.7 可观测性字段

- `current_node`: 当前节点

`graph_trace` 是状态查询接口展示进度的重要依据，也用于服务层估算任务进度。

## 6. 分析师节点工作方式

每个分析师节点的通用模式是：

```text
检查 state 中是否已有足够数据
  -> 如需工具，生成 AIMessage.tool_calls
  -> 条件路由进入 tools_xxx
  -> 工具节点执行并合并结果到 state
  -> 回到分析师节点
  -> 生成报告和 analyst_conclusions
  -> Msg Clear 清理消息上下文
  -> 下一个分析师
```

分析师节点不是简单地一次性输出结论，而是先决定是否需要工具数据。工具执行完成后，节点再次运行，基于工具结果写报告。

### 6.1 VIN Context Analyst

文件：

```text
vehicleagents/agents/analysts/vin_context_analyst.py
```

职责：

- 根据 VIN 获取车辆档案
- 获取历史 DTC
- 获取维修保养历史
- 判断车辆历史档案中是否已有故障线索

默认工具：

- `get_vehicle_profile_by_vin`
- `get_dtc_history_by_vin`
- `get_maintenance_history_by_vin`

默认 fallback 行为：

- 如果有 VIN 且没有工具结果，会默认调用车辆档案和 DTC 历史工具。
- 即使维护历史没被默认调用，报告里也会读取 state 中已有的 `maintenance_history`。
- 如果 VIN 缺失，会在 `data_gaps` 中记录 `vin_missing`，但不会中断流程。

输出：

- `vehicle_profile_report`
- 更新 `vehicle`
- 更新 `dtc_history`
- 更新 `maintenance_history`

### 6.2 Symptom Analyst

文件：

```text
vehicleagents/agents/analysts/symptom_analyst.py
```

职责：

- 分析用户描述的症状数量、最高严重程度、主诉
- 根据症状、DTC、车辆信息检索类似维修案例

默认工具：

- `retrieve_repair_cases`

默认 fallback 行为：

- 如果存在症状但 LLM 没有可用工具决策，会默认调用案例检索。
- 如果没有症状，仍会生成“缺少明确故障现象”的报告，不中断流程。

输出：

- `symptom_report`
- `similar_cases`
- `primary_complaint`

### 6.3 Diagnostic Code Analyst

文件：

```text
vehicleagents/agents/analysts/diagnostic_code_analyst.py
```

职责：

- 合并用户提交的 DTC 和历史 DTC
- 查询单个 DTC 的含义
- 查询 DTC 组合模式
- 输出故障码层面的证据判断

默认工具：

- `lookup_dtc_code`
- `search_dtc_combinations`

默认 fallback 行为：

- 如果有 DTC 且 LLM 没做工具决策，会默认对前两个 DTC 调用 `lookup_dtc_code`。
- 如果 DTC 数量少于 2，会补充调用 `search_dtc_combinations`。
- 即使工具失败，仍会基于 codes 生成 DTC 报告，并把错误放入 `tool_errors`。

输出：

- `dtc_report`
- 更新标准化后的 `dtc_codes`
- `lookups`
- `combinations`

### 6.4 Telemetry Analyst

文件：

```text
vehicleagents/agents/analysts/telemetry_analyst.py
```

职责：

- 获取传感器快照
- 获取时序信号
- 获取事件日志
- 执行确定性遥测规则检查

默认工具：

- `get_sensor_snapshot_by_vin`
- `get_sensor_timeseries_by_vin`
- `get_event_logs_by_vin`
- `analyze_telemetry_rules`

默认信号：

```text
rpm
battery_voltage
coolant_temp_c
stft_b1
ltft_b1
misfire_count_cyl_1
```

默认 fallback 行为：

- 如果请求中已有 snapshot 或 events，会优先调用 `analyze_telemetry_rules` 做规则判断。
- 如果只有 VIN，会依次拉取 snapshot、timeseries、event logs。
- 受工具调用上限影响，默认每轮最多调用有限数量的工具。
- 没有遥测异常时也会输出报告，结论为未发现明确异常。

确定性规则示例：

- `stft_b1 > 15`: 短期燃油修正偏高
- `ltft_b1 > 10`: 长期燃油修正偏高
- `misfire_count_cyl_1 > 0`: 1 缸失火计数非零
- `battery_voltage < 12.0`: 电压偏低
- 事件名为 `rough_idle_detected`: 怠速抖动事件与症状相关

输出：

- `telemetry_report`
- 更新 `sensor_snapshot`
- 更新 `sensor_timeseries`
- 更新 `event_logs`
- `rule_findings`

### 6.5 Knowledge Analyst

文件：

```text
vehicleagents/agents/analysts/knowledge_analyst.py
```

职责：

- 汇总车辆、症状、DTC、遥测报告
- 检索类似维修案例
- 输出知识库/案例层面的辅助证据

默认工具：

- `retrieve_repair_cases`

默认 fallback 行为：

- 如果没有 LLM 工具决策，会默认调用维修案例检索。
- 如果没有命中案例，也会生成报告并标记知识库未命中。

输出：

- `knowledge_report`
- `similar_cases`
- `case_count`
- `knowledge_sources`

## 7. 工具节点工作流程

工具节点由 `create_vehicle_tool_node()` 创建。

每个分析师对应一个工具节点：

| 分析师 | 工具节点 |
| --- | --- |
| VIN Context Analyst | `tools_vin_context` |
| Symptom Analyst | `tools_symptom` |
| Diagnostic Code Analyst | `tools_dtc` |
| Telemetry Analyst | `tools_telemetry` |
| Knowledge Analyst | `tools_knowledge` |

工具节点核心流程：

1. 读取上一条消息中的 `tool_calls`。
2. 根据当前分析师剩余工具调用额度截断可执行工具。
3. 超额工具不执行，记录到 `tool_errors`。
4. 对每个可执行工具调用 `_invoke_tool_with_retries()`。
5. 工具结果尝试 JSON 解析。
6. 将结果写入 `analyst_tool_results[analyst_key]`。
7. 对部分工具结果进行 state 合并，例如车辆档案、DTC 历史、传感器快照。
8. 生成 `ToolMessage` 返回给分析师节点。
9. 更新当前分析师的工具调用计数。
10. 写入 `graph_trace`。

### 7.1 工具调用限制

工具调用上限来自：

- 全局 `max_tool_calls`
- 或按分析师设置的 `analyst_max_tool_calls`

默认配置：

```python
{
  "vin_context": 2,
  "symptom": 1,
  "dtc": 4,
  "telemetry": 3,
  "knowledge": 1
}
```

如果工具调用超限：

- 超额调用不会执行
- 记录错误：

```json
{
  "analyst": "dtc",
  "tool": "...",
  "args": {},
  "error": "max_tool_calls exceeded for dtc"
}
```

Graph 不会失败，而是继续后续流程。

### 7.2 工具超时与重试

工具调用通过线程池执行，并支持：

- `tool_timeout_seconds`
- `tool_max_retries`

逻辑：

```text
attempts = max_retries + 1
每次尝试:
  -> 在线程池中调用 tool.invoke(args)
  -> 如果超时，取消 future
  -> 如果异常，记录 last_error
全部失败:
  -> 抛出最后一次错误
```

工具节点捕获错误后不会让 Graph 崩溃，而是：

- 写入 `tool_errors`
- 写入 `analyst_tool_results`，其中 `ok=false`
- 返回一个包含错误信息的 `ToolMessage`
- 继续回到分析师节点，让分析师基于现有证据输出降级报告

这是项目中非常重要的 fallback 设计：工具失败不等于诊断失败。

### 7.3 工具结果合并

部分工具返回结果会自动合并进 state：

| 工具 | 合并目标 |
| --- | --- |
| `get_vehicle_profile_by_vin` | `vehicle` |
| `get_dtc_history_by_vin` | `dtc_history` |
| `get_maintenance_history_by_vin` | `maintenance_history` |
| `get_sensor_snapshot_by_vin` | `sensor_snapshot` |
| `get_sensor_timeseries_by_vin` | `sensor_timeseries` |
| `get_event_logs_by_vin` | `event_logs` |

其他工具结果则主要保存在：

```text
analyst_tool_results
```

供对应分析师生成报告时读取。

## 8. 条件路由逻辑

条件路由定义在 `vehicleagents/graph/conditional_logic.py`。

### 8.1 分析师到工具节点

每个分析师节点执行后，条件逻辑会判断：

```text
如果最后一条 message 有 tool_calls
且当前分析师工具调用次数 < 最大次数
  -> 进入 tools_xxx
否则
  -> 进入 Msg Clear xxx
```

工具节点执行后会回到同一个分析师节点。分析师再次运行，看到已有工具结果后生成报告。

### 8.2 消息清理节点

每个分析师后都有一个：

```text
Msg Clear xxx
```

其作用是：

1. 删除 LangGraph messages 中之前的消息。
2. 将当前所有分析师压缩结论整理为一条新的 HumanMessage。
3. 避免上下文无限增长。
4. 让后续节点只看到简洁的分析摘要。

这对多节点 Agent 很关键：既保留跨节点信息，又控制上下文膨胀。

### 8.3 研究员辩论路由

辩论由两个节点构成：

- `Hypothesis Researcher`
- `Counterfactual Researcher`

路由逻辑：

```text
如果 debate count >= 2 * max_debate_rounds
  -> Diagnostic Planner
否则
  -> 在 Hypothesis 和 Counterfactual 之间交替
```

例如：

| `max_debate_rounds` | 节点执行次数 |
| --- | --- |
| `0` | 跳过辩论或很快进入 Planner |
| `1` | Hypothesis + Counterfactual，共 2 次 |
| `2` | Hypothesis + Counterfactual 循环 2 轮，共 4 次 |

### 8.4 安全评审路由

安全链路包括：

```text
Repair Advisor -> Safety Analyst -> Safety Judge
```

条件逻辑：

```text
如果 safety count >= max_safety_discuss_rounds
  -> Safety Judge
否则
  -> Repair Advisor
```

当前实现中 `Safety Analyst` 会增加 `safety_review_state.count`，然后路由到 `Safety Judge` 或回到 `Repair Advisor`。

## 9. 研究、规划、维修与安全节点

### 9.1 Hypothesis Researcher

文件：

```text
vehicleagents/agents/researchers/hypothesis_researcher.py
```

职责是提出故障假设。

当前是规则实现：

- 如果存在 `P0301`，提出 1 缸点火、火花塞、喷油器或压缩问题。
- 如果存在 `P0171`，提出进气泄漏、MAF、燃油供给导致的混合气过稀。
- 如果证据不足，则提示从症状引导检查开始。

输出写入：

- `diagnostic_debate_state.hypothesis_history`
- `diagnostic_debate_state.history`
- `diagnostic_debate_state.current_response`
- `diagnostic_debate_state.count`

### 9.2 Counterfactual Researcher

文件：

```text
vehicleagents/agents/researchers/counterfactual_researcher.py
```

职责是反证和风险提醒，避免“看到一个故障码就直接换件”。

当前规则输出包括：

- 不应仅凭一个 DTC 更换零件
- 在最终维修前确认点火、燃油、漏气、压缩证据
- 检查遥测和事件时间是否与症状匹配

输出写入：

- `diagnostic_debate_state.counterfactual_history`
- `diagnostic_debate_state.history`
- `diagnostic_debate_state.current_response`
- `diagnostic_debate_state.count`

### 9.3 Diagnostic Planner

文件：

```text
vehicleagents/agents/planner/diagnostic_planner.py
```

职责是把多分析师报告、辩论历史和相似记忆案例综合成诊断计划。

Planner 有两条路径：

```text
有 deep LLM
  -> 尝试 LLM 规划
  -> 解析失败/字段缺失时 fallback 到规则规划
无 deep LLM
  -> 直接规则规划
```

#### 9.3.1 Memory 读取

Planner 会构造当前 situation：

```json
{
  "vehicle": "...",
  "symptoms": "...",
  "dtc_codes": "...",
  "telemetry_report": "..."
}
```

然后调用：

```python
memory.get_memories(situation, n_matches=3)
```

获得最多 3 条相似历史维修案例，作为规划依据。

如果 memory 不存在、没有 `get_memories` 方法，或查询异常，返回空列表，不影响诊断。

#### 9.3.2 LLM 规划

如果启用了 LLM，Planner 会要求模型返回严格 JSON，至少包含：

- `ranked_hypotheses`
- `next_tests`
- `stop_conditions`
- `confidence`

如果 LLM 响应：

- 不是合法 JSON
- 不是 dict
- 缺少关键字段
- 调用抛异常

则返回 `None`，自动进入规则规划。

#### 9.3.3 规则规划

规则规划示例：

- 有 `P0301`：生成 “Cylinder 1 ignition system fault”
- 有 `P0171`：生成 “Lean mixture from intake leak or fuel delivery issue”
- 如果没有明确 DTC：生成 “Undetermined fault”

计划还会包含：

- `ranked_hypotheses`
- `next_tests`
- `stop_conditions`
- `confidence`
- `planner_source`
- `similar_memory_cases`

### 9.4 Repair Advisor

文件：

```text
vehicleagents/agents/advisor/repair_advisor.py
```

职责是生成维修建议。

当前实现是规则建议：

- 车辆可驾驶性：`limited`
- 建议避免高负载驾驶
- 先做低成本检查再换件
- MIL 闪烁时停止驾驶并安排维修
- 估算费用区间：CNY 200 到 1500

输出写入：

```text
repair_advice
```

### 9.5 Safety Analyst

文件：

```text
vehicleagents/agents/safety/safety_analyst.py
```

职责是根据故障码评估驾驶安全风险。

当前规则：

- 如果 DTC 包含 `P0300` 或 `P0301`，安全等级为 `medium`
- 否则为 `low`

同时提醒：

- MIL 闪烁
- 振动加剧

## 10. Memory 记忆机制

Memory 实现在：

```text
vehicleagents/agents/utils/memory.py
```

项目定义了统一接口：

```python
class VehicleMemoryStore(Protocol):
    def add_case(self, situation: str, recommendation: str, metadata: dict | None = None) -> None:
        ...

    def search(self, current_situation: str, n_matches: int = 3) -> list[dict]:
        ...
```

上层统一通过：

```python
VehicleDiagnosisMemory
```

访问，不关心底层是内存、SQLite 还是 MongoDB。

### 10.1 支持的 Memory Store

| Store | 使用场景 | 持久化 |
| --- | --- | --- |
| `InMemoryVehicleMemoryStore` | 默认轻量运行、单元测试、未注入存储时 | 否 |
| `SQLiteVehicleMemoryStore` | 测试或 SQLite Repository 注入场景 | 是 |
| `MongoVehicleMemoryStore` | 默认服务运行场景 | 是 |

### 10.2 Service 如何选择 Memory

在 `VehicleDiagnosisService._build_memory()` 中：

```text
如果 repository 有 db_path
  -> 使用 SQLiteVehicleMemoryStore(db_path)
否则
  -> 使用 MongoVehicleMemoryStore(mongo_uri, mongo_database)
```

因此：

- 测试注入 SQLite Repository 时，memory 和任务共用 SQLite 文件。
- 生产默认 Mongo Repository 时，memory 写入 MongoDB。

MongoDB memory 集合：

```text
vehicle_memory_cases
```

SQLite memory 表：

```text
vehicle_memory_cases
```

### 10.3 Memory 写入时机

主要写入时机是用户回填维修结果：

```http
POST /api/vehicle-diagnosis/tasks/{task_id}/outcome
```

Service 会构造：

```json
{
  "situation": {
    "vin": "...",
    "vehicle": "...",
    "symptoms": "...",
    "dtc_codes": "...",
    "telemetry_findings": "..."
  },
  "recommendation": {
    "confirmed_root_cause": "...",
    "repairs_performed": "...",
    "resolved": true
  },
  "metadata": {
    "task_id": "...",
    "stored_at": "..."
  }
}
```

然后调用：

```python
self.memory.add_case(...)
```

这意味着系统记忆不是盲目保存模型输出，而是保存“用户或维修流程确认后的真实结果”。这更适合诊断场景，因为确认根因比 Agent 猜测更有价值。

### 10.4 Memory 读取时机

读取发生在 `Diagnostic Planner`。

Planner 会将当前问题的车辆、症状、故障码和遥测报告序列化成 situation，然后检索相似历史案例：

```python
memory.get_memories(situation, n_matches=3)
```

返回的相似案例会进入：

```text
diagnostic_plan.similar_memory_cases
```

并参与故障假设概率调整。例如规则规划中，存在相似 memory 时，`P0301` 的点火系统故障概率会略高。

### 10.5 Memory 检索算法

当前检索不是向量数据库，而是轻量 token overlap：

1. 将 query 用正则拆成 token。
2. 将每条 memory 的 situation + recommendation 拆成 token。
3. 计算 token 交集数量。
4. 得分：

```text
score = overlap / max(len(query_tokens), 1)
```

5. 先按 score 排序，再按新旧顺序辅助排序。
6. 返回前 `n_matches` 条。

这个实现简单、可测试、无外部模型依赖，后续可以替换成向量检索，不需要改 Agent 节点调用方式。

## 11. 数据源与 Provider Fallback

数据统一入口在：

```text
vehicleagents/dataflows/interface.py
```

它对外暴露：

- `get_vehicle_profile_by_vin`
- `get_sensor_snapshot_by_vin`
- `get_sensor_timeseries_by_vin`
- `get_event_logs_by_vin`
- `get_dtc_history_by_vin`
- `get_maintenance_history_by_vin`
- `lookup_dtc_code`
- `search_dtc_combinations`
- `retrieve_repair_cases`

### 11.1 数据源模式

通过环境变量控制：

```env
VEHICLE_DIAGNOSIS_DATA_PROVIDER=auto
```

支持：

| 模式 | 行为 |
| --- | --- |
| `mock` | 不连接 MongoDB，直接使用内置 mock 数据 |
| `mongo` | 强制连接 MongoDB，连接失败或查询异常会抛错 |
| `auto` | 优先使用 MongoDB，连接失败、依赖缺失或查询为空时 fallback 到 mock/local 数据 |

### 11.2 Mongo 连接 fallback

`mongo_client.get_database()` 行为：

```text
mode == mock
  -> 返回 None
mode == mongo
  -> Mongo 依赖缺失/连接失败时抛异常
mode == auto
  -> Mongo 依赖缺失/连接失败时返回 None
```

当返回 None 时，各 provider 会走本地 mock。

### 11.3 查询异常 fallback

`safe_find_one()` 和 `safe_find_many()` 封装 Mongo 查询。

```text
mode == mongo
  -> 查询异常抛出
mode == auto/mock
  -> 查询异常返回空 dict/list
```

这让开发和演示环境更稳：Mongo 不可用时不会直接导致整个诊断失败。

### 11.4 VIN Provider fallback

文件：

```text
vehicleagents/dataflows/providers/vin_database.py
```

行为：

- 有 VIN 且 Mongo 有数据：返回 Mongo 数据，标记 `source=mongo_vehicle_profiles`
- Mongo 无数据或不可用：返回 mock 车辆档案
- 历史 DTC 无数据时返回 mock `P0301`
- 维护历史无数据时返回 mock 维护记录

### 11.5 Sensor Provider fallback

文件：

```text
vehicleagents/dataflows/providers/sensor_database.py
```

行为：

- Mongo 有最新 snapshot：返回 Mongo snapshot，标记 source
- 否则返回 mock snapshot，包括 rpm、电压、水温、燃油修正、1 缸失火计数
- 时序数据优先查 Mongo，否则为请求的每个 signal 返回 mock 点位

### 11.6 Event Provider fallback

文件：

```text
vehicleagents/dataflows/providers/event_database.py
```

行为：

### 11.7 Repair Case Provider fallback

```text
vehicleagents/dataflows/providers/local_cases.py
```

行为：

### 11.8 DTC Provider

文件：

当前 DTC 字典是本地静态数据，支持：

- `P0300`
- `P0301`
- `P0171`
- `P0420`

未知故障码会返回：

```json
  "code": "...",
```

DTC 组合规则包括：

## 12. LLM Fallback

项目支持 OpenAI-compatible `/chat/completions` API。

LLM 配置加载流程：

1. 读取 `config/llm_config.json`。
2. 再用环境变量覆盖 JSON 配置。
3. 如果 `VEHICLE_DIAGNOSIS_LLM_ENABLED=false`，则 `quick_llm` 和 `deep_llm` 都为 `None`。
4. 如果开启，则创建两个 `OpenAICompatibleLLM`：
   - quick：用于轻量判断、工具决策
   - deep：用于深度规划

### 12.1 分析师工具决策 fallback

分析师节点会先尝试：

```python
invoke_llm_tool_decision(...)
```

如果 LLM：

- 未启用
- 调用异常
- 返回格式不能解析
- 没有产生 tool_calls

分析师会使用硬编码默认工具策略。

示例：

- DTC 分析师默认查故障码字典
- 症状分析师默认检索维修案例
- 遥测分析师默认拉传感器数据或跑规则

### 12.2 Planner fallback

Planner 使用 deep LLM 时，如果任何步骤失败，会 fallback 到 `_deterministic_plan()`。

失败包括：

- API 异常
- 返回不是 JSON
- JSON 不包含 `ranked_hypotheses`
- JSON 不包含 `next_tests`

因此即使不配置 API Key，系统仍然能完成规则诊断。

### 12.3 LLM API Key 缺失

如果启用 LLM 但对应 API Key 不存在，`OpenAICompatibleLLM.invoke()` 会抛出：

```text
missing API key environment variable
```

对于分析师工具决策，该异常会被吞掉并 fallback 到默认工具策略。

对于 Planner，该异常会被 `_try_llm_plan()` 捕获并 fallback 到规则规划。

## 15. 可观测性与稳定性设计

### 15.1 请求级日志

`RequestLoggingMiddleware` 会：

- 读取或生成 `x-request-id`
- 记录 method、path、status code、耗时
- 将 `x-request-id` 写回响应头

### 15.2 Redis 限流

`RedisRateLimitMiddleware` 基于：

```text
rate_limit:{client}:{path}
```

做请求计数。

超过阈值返回：

```http
429 Too Many Requests
```

`/health` 不参与限流。

### 15.3 Graph trace

所有关键节点通过 `append_trace()` 写入：

```json
{
  "node": "...",
  "status": "...",
  "detail": "..."
}
```

这让前端或 API 调用方可以看到 Agent 当前运行到哪一步，而不是只能等待最终结果。

### 15.4 工具错误显式返回

工具错误不会被静默吞掉，而是进入：

```text
tool_errors
```

最终结果中也会包含 `tool_errors`，便于排查数据源或工具稳定性问题。

### 15.5 中断任务恢复

服务启动时会查询状态为：

- `submitted`
- `queued`
- `running`

的历史任务，将它们重置为 `submitted` 并重新入队。这可以处理服务重启时 worker 中断的问题。

## 16. Fallback 汇总

项目中的 fallback 可以归纳为以下几类：

| 类型 | 触发条件 | Fallback |
| --- | --- | --- |
| 数据源 fallback | `auto` 模式下 Mongo 不可用或无数据 | 使用 mock/local 数据 |
| 查询异常 fallback | `auto` 模式下 Mongo 查询失败 | 返回空结果，由 provider 再 fallback |
| 工具决策 fallback | LLM 未启用、异常或无 tool_calls | 分析师使用默认工具调用策略 |
| 工具执行 fallback | 工具超时/异常/超限 | 记录 `tool_errors`，继续诊断 |
| 工具重试 fallback | 工具临时失败 | 按 `tool_max_retries` 重试 |
| Planner fallback | LLM 规划失败或返回不合格 JSON | 使用规则诊断计划 |
| Memory fallback | memory 不存在或搜索失败 | 返回空相似案例，继续规划 |
| 结果 fallback | 未生成 `structured_result` | 返回基础 `diagnosis_id + summary` |
| 任务恢复 fallback | 服务重启后存在未完成任务 | 重置并重新入队 |
| 取消 fallback | 任务运行中无法强杀线程 | 协作式取消，Graph 进度回调检查状态 |

## 17. 一次典型诊断示例

请求：

```json
{
  "vin": "LFV3A23C0J3000001",
  "symptoms": [
    {
      "name": "rough idle",
      "severity": "medium"
    }
  ],
  "dtc_codes": ["P0301", "P0171"],
  "parameters": {
    "selected_analysts": ["vin_context", "symptom", "dtc", "telemetry", "knowledge"],
    "diagnosis_depth": "standard"
  }
}
```

可能执行过程：

1. API 创建任务，状态 `queued`。
2. worker 取任务，状态 `running`。
3. VIN 分析师根据 VIN 拉取车辆档案和历史 DTC。
4. 症状分析师根据 rough idle 检索类似维修案例。
5. DTC 分析师合并 `P0301`, `P0171` 和历史 DTC，查询故障码含义。
6. 遥测分析师拉取传感器快照、时序、事件日志，并执行规则检查。
7. 知识分析师基于车辆、症状、DTC、遥测报告检索维修案例。
8. Hypothesis Researcher 提出：1 缸点火系统故障、混合气过稀等假设。
9. Counterfactual Researcher 提醒：不要只凭故障码换件，要验证点火、燃油、漏气和压缩。
10. Diagnostic Planner 读取相似 memory，尝试 LLM 规划；如不可用则规则规划。
11. Repair Advisor 给出驾驶限制和检查建议。
12. Safety Analyst 判断失火相关故障码带来中等安全风险。
13. Safety Judge 生成结构化最终诊断。
14. Service 保存 result，状态 `completed`。
15. 用户完成真实维修后通过 outcome 接口回填确认根因。
16. Memory 保存该确认案例，未来相似故障会被 Planner 检索到。

## 18. 简历表达时可提炼的技术点

这部分是从工程实现中提炼出来的简历素材，不是夸张包装。

可以突出：

- 设计并实现基于 FastAPI + LangGraph 的车辆故障诊断多 Agent 后端系统。
- 构建 VIN、症状、DTC、遥测、知识库五类分析师节点，形成“数据采集 -> 证据分析 -> 假设辩论 -> 诊断规划 -> 安全评审”的可解释诊断链路。
- 封装车辆档案、故障码、传感器、事件日志、维修案例等工具，并支持工具超时、重试、调用上限和错误降级。
- 设计 MongoDB/SQLite 可切换的任务持久化与 Memory 存储，支持用户回填真实维修结果形成案例记忆。
- 实现 Redis 异步任务队列、worker 并发、任务进度 streaming、任务取消、超时控制和服务重启后的任务恢复。
- 支持 OpenAI-compatible LLM 接入，并在工具决策和诊断规划阶段提供规则 fallback，保证无 LLM 或 LLM 异常时系统仍可运行。
- 通过 `graph_trace`、`tool_errors`、`analyst_conclusions` 等字段增强 Agent 运行过程可观测性和可解释性。

## 19. 当前实现的边界

项目当前是一个可运行的后端 MVP/工程原型，仍有可继续增强的地方：

- 研究员、维修顾问、安全评审大多是规则实现，后续可接入更完整的 LLM 推理提示词。
- Memory 当前使用 token overlap 检索，后续可升级为 embedding/vector search。
- 数据源 provider 已支持 Mongo 和 mock fallback，但生产环境仍需要接入真实 VIN、OBD、TSP 或维修知识库。
- 尚未加入用户认证和多租户隔离。
- 尚未加入 Prometheus 指标、成本统计、token 统计和更完整的审计日志。

## 20. 一句话总结

这个项目不是一个简单的“调用大模型问答接口”，而是一个工程化的车辆诊断 Agent 后端：它把诊断任务异步化，把车辆数据工具化，把多角色推理图谱化，把中间过程可观测化，并通过维修结果回填形成可复用的案例记忆；同时在 LLM、工具和数据源不可用时都有规则或 mock fallback，保证系统具备可演示、可测试和可扩展的稳定诊断链路。
