# 项目说明文档

本项目是一个 FastAPI + LangGraph 的车辆故障诊断服务。当前默认流程按新的职责边界拆成 4 个前置 Agent 和 1 个最终汇总 Agent：

1. 历史故障和维修记录 Agent
2. 用户主观描述 Agent
3. DTC Agent
4. RAG 知识库 Agent
5. 汇总/辩论 Agent

## 支持的 selected_analysts

```text
vin_context
symptom
dtc
knowledge
experience
```

默认启用：

```text
vin_context, symptom, dtc, knowledge
```

`experience` 是可选兼容节点，用于查询 outcome 确认后的维修记忆；默认流程中，历史案例主要由 `knowledge` 作为 RAG 资料的一部分检索。

## 当前图流程

```text
START
  -> VIN Context Analyst
  -> Symptom Analyst
  -> Diagnostic Code Analyst
  -> Knowledge Analyst
  -> Summary Agent
  -> END
```

带工具的 Agent 会先请求封装好的 Skill/Tool，再回到对应 Agent 生成报告。LLM 不直接写 SQL。

## Agent 边界

| Agent | 职责 | 不做什么 |
| --- | --- | --- |
| `vin_context` | 通过 VIN 查询车辆档案、历史 DTC、维修/保养记录 | 不推断最终根因 |
| `symptom` | 语义标准化用户主诉，生成疑似故障点和排查方向 | 不查 RAG，不做最终诊断 |
| `dtc` | 按排查方向收集 DTC、组合模式、冻结帧等客观事实 | 不做主客观交叉验证 |
| `knowledge` | 查询车辆手册、维修资料、历史案例 | 不裁决谁对谁错 |
| `summary` | 统一辩论和交叉验证，输出最终诊断 | 不写入经验库 |

## 保留工具

| 工具 | 用途 |
| --- | --- |
| `get_vehicle_profile_by_vin` | 根据 VIN 查询车辆档案和里程 |
| `get_dtc_history_by_vin` | 根据 VIN 查询历史故障码 |
| `get_maintenance_history_by_vin` | 根据 VIN 查询维修/保养记录 |
| `lookup_dtc_code` | 查询单个 DTC 含义 |
| `search_dtc_combinations` | 查询故障码组合模式 |
| `retrieve_repair_cases` | 查询知识库/案例库 |

## 最终结果

最终由 `Summary Agent` 写入 `structured_result`。主要字段：

- `summary`
- `final_conclusion`
- `confidence_score`
- `confidence_level`
- `ranked_hypotheses`
- `debate_notes`
- `reasoning_process`
- `inspection_plan`
- `analyst_conclusions`
- `reports`
- `tool_errors`
- `diagnostic_trace`

## 经验库

维修完成并确认真实根因后，通过：

```text
POST /api/vehicle-diagnosis/tasks/{task_id}/outcome
```

写入 memory。系统不会把未确认的 Summary 推断自动写入经验库。
