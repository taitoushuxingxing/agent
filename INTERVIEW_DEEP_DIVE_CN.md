# 车辆故障诊断 Agent 项目面试深挖问答

这份文档用于面试准备。核心目标不是把项目包装得很玄，而是把它讲成一个可落地、可扩展、可排错的“车辆诊断证据融合后端”。

一句话定位：

> 这个项目不是让大模型凭空修车，而是把车辆诊断拆成几类证据采集任务，再由 Summary Agent 做最终证据融合和冲突仲裁。

## 1. 项目到底解决了什么问题？

**问题：** 这个项目听起来像多个 Agent 套壳，它到底解决了什么实际问题？

**回答：**

它解决的是车辆故障诊断中“多源证据难融合”的问题。真实诊断不是只看用户一句话，也不是只看一个故障码，而是要同时看：

- 用户主诉：车主感受到什么
- VIN 历史：这台车过去发生过什么
- DTC 数据：车辆客观报了什么故障
- 维修资料：手册、案例、技术公告怎么解释

所以我没有让一个大模型直接猜，而是把它拆成几个职责明确的证据采集节点，最后由 Summary Agent 做统一判断。这样能降低幻觉，也方便定位是哪一类证据出了问题。

## 2. 为什么不用一个大 Prompt 直接诊断？

**问题：** 既然最后还是大模型总结，为什么不直接一个 Prompt 全部搞定？

**回答：**

一个 Prompt 可以做 Demo，但工程上有几个问题：

- 职责混乱：主诉理解、数据库查询、DTC 解释、RAG 检索、最终判断都挤在一起。
- 不可观测：结果错了，很难知道错在主诉解析、DTC 查询还是 RAG 检索。
- 不安全：如果让模型直接写 SQL 或拼查询语句，存在误查、慢查、甚至危险操作风险。
- 不可替换：后续想把 mock DTC 换成真实 OBD/TSP、把本地案例换成 MCP RAG，会牵一发而动全身。

多 Agent 拆分后，每个节点只做一件事，输入输出结构化，方便测试、替换和排错。

## 3. 这个项目现在是真诊断还是架构 MVP？

**问题：** 你这个项目真的能诊断车辆故障吗？

**回答：**

我会把它定义为车辆诊断后端的架构 MVP。它已经跑通了完整链路：

- FastAPI 任务接口
- LangGraph 多节点流程
- VIN/DTC/案例 Provider 抽象
- Tool 调用、超时、重试和错误记录
- Summary 结构化输出
- outcome 确认后写入经验库
- 自动化测试覆盖主流程

但当前底层数据主要还是 mock/本地字典级别，没有接真实 OBD、TSP 或大规模维修知识库。所以它的重点不是“现在知识库多全”，而是“诊断链路、Agent 边界、证据融合方式”是按生产系统思路设计的。

## 4. 每个 Agent 的职责是什么？

**问题：** 你怎么解释每个 Agent 的分工？

**回答：**

我按证据类型拆分：

- `vin_context`：历史事实收集者。根据 VIN 查车辆档案、历史 DTC、维修/保养记录。
- `symptom`：主观线索翻译器。把用户口语化描述标准化，输出疑似故障点和排查方向。
- `dtc`：客观取证者。根据排查方向查询 DTC、组合模式、冻结帧和数据流事实。
- `knowledge`：维修资料检索者。查车辆手册、技术公告、召回通病、历史案例。
- `summary`：最终裁决者。统一做交叉验证、冲突仲裁、置信度和检查计划。

关键点是：前置 Agent 不抢 Summary 的工作，只提交自己负责的证据。

## 5. 主观描述 Agent 为什么重要？

**问题：** 用户主观描述 Agent 不就是把文本总结一下吗？

**回答：**

它的价值是把“人话”翻译成“车话”。比如用户说“车子一加油就顿一下”，这句话不适合直接给 DTC 或 RAG 用。主观描述 Agent 会把它变成结构化信息：

- 触发条件：加速时
- 标准症状：顿挫/动力中断
- 疑似系统：发动机点火、燃油供给、变速箱换挡控制
- 建议 DTC 关注：P0300-P0306、燃油修正、变速箱控制码
- 建议 RAG 查询词：车型 + 加速顿挫 + 通病/维修案例

它有两个下游输出：

- 语义标准化结果给 RAG 做检索输入。
- 排查方向给 DTC Agent 做定向查询。

## 6. DTC Agent 为什么不直接做交叉验证？

**问题：** DTC Agent 已经拿到了故障码，为什么不在这里判断是不是和用户主诉匹配？

**回答：**

因为 DTC Agent 的职责应该保持纯粹：只做客观取证。它可以解释 DTC 含义，但不负责最终裁决。

如果 DTC Agent 提前做交叉验证，容易有两个问题：

- 过早过滤：一些看起来无关的故障码，可能结合 RAG 后才发现有关。
- 职责膨胀：DTC 既查数据又判断，又和 RAG/Summary 的职责重叠。

所以我把交叉验证统一放到 Summary Agent。DTC 只提交 DTC、冻结帧、数据流等事实。

## 7. RAG Agent 的输入是什么？

**问题：** RAG Agent 是只拿用户问题去查，还是拿 DTC 去查？

**回答：**

最好同时拿两类信息：

- 主观描述 Agent 的语义标准化结果和疑似故障点
- DTC Agent 查到的客观故障码、冻结帧、数据流事实

这样 RAG 的查询更精准。例如不是简单查“冷车抖动”，而是查：

> 该车型 + 冷车抖动 + P0301 + 失火 + 维修案例/手册步骤

这比单独用用户原始描述检索可靠很多。

## 8. Summary Agent 最后拿到哪些证据？

**问题：** Summary 最后靠什么判断？

**回答：**

Summary 拿到四类材料：

1. VIN Agent 的历史背景：车辆档案、历史 DTC、维修记录。
2. Symptom Agent 的主观线索：语义标准化、疑似故障点、排查方向。
3. DTC Agent 的客观证据：DTC、故障码解释、冻结帧、数据流。
4. RAG Agent 的维修知识：手册、技术公告、历史案例、通病/召回。

然后 Summary 做统一辩论。例如用户感觉是变速箱顿挫，但 DTC 指向发动机失火，RAG 又说明失火会造成类似顿挫体感，那么 Summary 会倾向发动机点火系统，而不是变速箱。

## 9. Summary 的“辩论”是不是很虚？

**问题：** 你说 Summary 辩论，听起来像 Prompt 玄学，怎么保证靠谱？

**回答：**

我这里的“辩论”不是让模型自由发挥，而是给它一个结构化案件卷宗和明确规则：

- 只能使用上游 Agent 的报告。
- 客观数据优先于主观体感。
- 最终结论尽量需要 DTC 或 RAG 支撑。
- 与本次主诉无关的历史偶发码要降权。
- 缺少证据时必须标为疑似，而不是强行下结论。

同时结果是结构化 JSON，包括 `ranked_hypotheses`、`evidence_for`、`evidence_against`、`confidence_score` 和 `inspection_plan`，便于前端展示和后续审计。

## 10. 数据库为什么不用大模型写 SQL？

**问题：** 为什么不让 LLM 根据自然语言生成 SQL？

**回答：**

车辆诊断系统里数据库查询属于高风险、强确定性的能力，不适合交给 LLM 自由生成 SQL。

我的设计是把数据库查询封装成 Skill/Tool：

- LLM 只传参数，比如 VIN、DTC code、查询范围。
- SQL/Mongo 查询由后端固定函数执行。
- Tool 内做参数校验、超时、重试和错误记录。

这样可以避免 SQL 注入、危险操作、全表扫描，也降低数据库 schema 变化对 Prompt 的影响。

## 11. 哪些 Agent 可以并行？

**问题：** 多 Agent 会不会很慢？哪些可以并行？

**回答：**

业务上最适合并行的是第一阶段：

```text
vin_context || symptom
```

因为：

- `vin_context` 只依赖 VIN。
- `symptom` 只依赖用户主诉。

后面依赖关系更强：

```text
symptom -> dtc -> knowledge -> summary
```

DTC 最好等 symptom 给出排查方向后执行，RAG 最好等 symptom 和 DTC 都完成后执行，这样检索最精准。

当前代码为了状态合并和 trace 简洁，采用顺序执行；后续可以用 LangGraph reducer/并行分支优化 `vin_context` 和 `symptom`。

## 12. 如果用户没有提供 DTC 怎么办？

**问题：** 没有 DTC 的情况下系统还能工作吗？

**回答：**

可以，但 Summary 会降低置信度。

流程会变成：

- Symptom Agent 根据主诉给出疑似方向。
- Knowledge Agent 根据语义标准化结果查手册和案例。
- Summary 输出“疑似故障”，建议下一步读取 DTC、路试或补充数据。

也就是说没有 DTC 时不会强行确诊，而是给出待验证假设和下一步检查计划。

## 13. 如果 DTC 和用户主诉冲突怎么办？

**问题：** 用户说空调不冷，但 DTC 有发动机失火码，你怎么处理？

**回答：**

这就是 Summary 的冲突仲裁场景。

它会看：

- DTC 是否当前故障还是历史故障。
- RAG 是否支持该 DTC 能解释用户主诉。
- VIN 历史中是否有类似问题。
- 用户主诉和 DTC 所属系统是否有因果关系。

如果没有关系，就把该 DTC 标为干扰项或历史遗留码，降低优先级。不会因为有 DTC 就机械地当成最终原因。

## 14. 如果 RAG 检索不到资料怎么办？

**问题：** RAG 没命中怎么办？

**回答：**

RAG 没命中不是失败，而是一种证据缺口。Summary 会：

- 保留 DTC 和主观线索的判断。
- 降低置信度。
- 在 `debate_notes` 或 `inspection_plan` 里说明缺少手册/案例支撑。
- 建议进一步读取数据、补充车型资料或人工复核。

这比让模型编一个维修方案更安全。

## 15. 经验库为什么不自动写？

**问题：** Summary 得到结论后为什么不自动写入经验库？

**回答：**

因为 Summary 输出的是诊断推断，不一定是真实根因。如果自动写入经验库，误诊会污染后续检索。

所以我设计为：

```text
诊断完成 -> 人工/业务系统确认真实维修结果 -> outcome API -> 写入经验库
```

只有确认过的维修闭环案例才进入经验库。

## 16. 这个项目的测试覆盖了什么？

**问题：** 你怎么保证流程是可用的？

**回答：**

测试覆盖了主要工程链路：

- LangGraph 主流程能跑通。
- selected_analysts 能生效。
- Tool 调用结果能写入 state。
- Tool 失败时系统不中断。
- Tool 支持重试。
- Tool 调用次数限制能生效。
- LLM Summary 可用时优先用 LLM，不可用时走规则 fallback。
- 服务层能创建、执行、查询、取消任务。
- outcome 能写入经验库。

所以它不是只写了几个 Prompt，而是一个后端工作流系统。

## 17. 现在最大的不足是什么？

**问题：** 你觉得这个项目目前最大的短板是什么？

**回答：**

主要有三个：

1. 底层数据还是 mock/本地字典，没有接真实 OBD、TSP 或维修系统。
2. RAG 目前是本地案例检索，后续需要接真正的车辆手册、内部维修文档和 MCP 知识库。
3. 当前默认流程是顺序执行，`vin_context` 和 `symptom` 后续可以并行优化。

但这些是数据源和性能层面的增强，不影响当前架构边界。

## 18. 如果接真实车端数据，你会怎么扩展？

**问题：** 如果要接真实 OBD/TSP，你怎么改？

**回答：**

我会保持 Agent 边界不变，只替换 Tool/provider：

- `lookup_dtc_code` 保留为 DTC 字典解释。
- 新增 `read_vehicle_dtcs` 读取当前/历史/pending DTC。
- 新增 `read_freeze_frame` 读取冻结帧。
- 新增 `read_data_stream` 读取关键数据流。
- Tool 内部处理协议、鉴权、超时、重试和数据清洗。

DTC Agent 仍然只拿工具结果生成客观证据报告，Summary 仍然负责最终裁决。

## 19. 如果接真实 RAG，你会怎么扩展？

**问题：** 如果要接 MCP RAG 或企业知识库，你怎么改？

**回答：**

我会替换 `retrieve_repair_cases` 的 provider 或新增 RAG Tool，接口保持类似：

```json
{
  "vehicle": {},
  "semantic_standardization": {},
  "suspected_fault_points": [],
  "dtc_codes": [],
  "retrieval_goal": "..."
}
```

底层可以接：

- 车辆维修手册
- 内部维修案例
- 技术公告 TSB
- 召回信息
- 工单系统

Knowledge Agent 的职责不变：只返回资料，不做最终判断。

## 20. 面试 30 秒讲法

**回答模板：**

这个项目我定位成车辆故障诊断的证据融合后端，而不是让大模型直接猜故障。用户提交 VIN、主诉和 DTC 后，系统通过 LangGraph 拆成几个职责明确的节点：VIN Agent 查历史记录，Symptom Agent 把用户口语标准化并给出排查方向，DTC Agent 查询客观故障码，RAG Agent 查维修手册和案例，最后 Summary Agent 统一做证据交叉验证和冲突仲裁。数据库查询都封装成 Tool，LLM 不直接写 SQL。这样做的好处是每一步可观测、可替换、可测试，也方便后续接真实 OBD、TSP 或 MCP 知识库。

## 21. 面试 2 分钟讲法

**回答模板：**

这个项目是一个 FastAPI + LangGraph 的车辆故障诊断后端。我没有让大模型直接根据用户一句话下结论，因为真实车辆诊断需要融合多类证据：用户主诉、车辆历史、DTC 客观数据、维修手册和历史案例。

所以我把流程拆成几个 Agent。`vin_context` 根据 VIN 查车辆档案、历史 DTC 和维修记录；`symptom` 负责把用户的口语化描述做语义标准化，输出疑似故障点和排查方向；`dtc` 根据这些方向查询和解释故障码、组合模式、冻结帧和数据流事实；`knowledge` 根据主观线索和 DTC 结果查维修手册、技术公告、通病和案例；最后 `summary` 统一做证据融合和冲突仲裁，输出最终故障排序、置信度和检查计划。

工程上我把数据库和底层查询都封装成 Tool，LLM 只负责参数提取和最终证据融合，不直接写 SQL。每个 Agent 的报告都会进入 state，方便观察和排错。现在底层数据主要是 mock 和本地字典，重点是跑通了可扩展的诊断链路；后续可以把 provider 替换成真实 OBD/TSP、企业维修库或 MCP RAG。

## 22. 被问懵时的兜底句

如果一时不知道怎么展开，可以回到这几句话：

- 这个项目的核心不是“Agent 多”，而是“证据类型拆分”。
- 前置 Agent 只采集和整理证据，不做最终裁决。
- Summary Agent 才负责主观、客观、知识库之间的交叉验证。
- 数据库查询通过 Tool 封装，LLM 不直接写 SQL。
- 当前是架构 MVP，数据源可以替换成真实 OBD/TSP/RAG。

## 23. 简历里“长推理链路和幻觉”怎么写？

**原始想表达：**

复杂故障诊断链路很长，LLM 容易丢上下文、混淆主观描述和客观证据，所以项目用 LangGraph 拆节点，并在 Summary 里做统一交叉校验。

**推荐简历表述：**

> 针对复杂故障诊断长链路易丢上下文、主客观证据易混淆的问题，基于 LangGraph 拆分 VIN 历史、主观症状、DTC 取证、RAG 检索与 Summary 裁决节点，实现结构化状态流转和统一交叉校验，降低单 Prompt 推理的幻觉与不可追踪风险。

**如果前一条已经写过 LangGraph，推荐用这个不重复版本：**

> 针对复杂故障诊断长链路易丢上下文、主客观证据易混淆的问题，设计结构化状态流转与证据隔离机制，并在 Summary Agent 中统一完成交叉校验和冲突仲裁，降低单 Prompt 推理的幻觉与不可追踪风险。

**最短版本：**

> 设计结构化状态管理与 Summary 交叉校验机制，降低复杂诊断长链路中的幻觉和不可追踪风险。

## 24. 什么叫长推理链路易丢上下文？

**问题：** 你说复杂故障诊断长推理链路容易中断或丢上下文，具体是什么意思？

**回答：**

如果用一个大 Prompt 从用户主诉一路推到最终根因，中间会经历主诉理解、VIN 历史查询、DTC 解释、RAG 检索、维修建议生成等多个步骤。链路一长，模型容易丢掉前面关键信息，也容易把用户主观感受当成客观事实。

比如用户说“像变速箱顿挫”，但实际 DTC 可能是发动机缺火。如果单 Prompt 直接判断，模型容易被用户表达带偏。我的做法是把链路拆成多个 LangGraph 节点，每个节点只产出自己负责的结构化报告，最后再统一融合。

## 25. 多节点状态管理具体做了什么？

**问题：** 你说基于 LangGraph 做多节点状态管理，具体管理了哪些状态？

**回答：**

我在共享 state 里维护每一类证据的独立字段，比如：

- `vehicle_profile_report`：VIN 历史和车辆档案
- `symptom_report`：主观描述语义标准化和疑似方向
- `dtc_report`：DTC、冻结帧、数据流等客观证据
- `knowledge_report`：RAG 检索到的维修资料和案例
- `analyst_tool_results`：各 Agent 的工具调用结果
- `tool_errors`：工具超时、失败、重试后的错误记录
- `graph_trace`：每个节点的执行轨迹

每个 Agent 只写自己的报告，不覆盖其他节点的信息。这样即使某个工具失败，系统也能保留已有证据，并让 Summary 输出保守结论，而不是整条链路直接失败。

## 26. 交叉校验机制到底怎么做？

**问题：** 你说 Summary Agent 做交叉校验和冲突仲裁，具体怎么校验？

**回答：**

我没有让 DTC Agent 或 RAG Agent 在中间提前下结论，而是让它们只提交证据。最后 Summary Agent 拿四类材料统一判断：

- VIN 历史记录
- 用户主观线索
- DTC 客观证据
- RAG 维修资料

比如用户感觉是变速箱顿挫，但 DTC 查到 `P0301` 一缸失火，RAG 资料说明发动机缺火会造成动力中断和顿挫体感，那么 Summary 会把最终方向裁决到点火系统，而不是机械相信用户的主观判断。

## 27. 这套机制怎么降低幻觉？

**问题：** 你说降低幻觉，依据是什么？是不是只是 Prompt 写得更长？

**回答：**

这里不是让模型“想得更久”，而是让模型“少做混杂任务”。我把主诉理解、DTC 取证、RAG 检索和最终判断拆开，每一步都有结构化输入输出。

降低幻觉风险主要靠三点：

- **证据隔离**：主观描述、客观 DTC、维修资料分别保存，避免混在一起。
- **工具约束**：数据库和 DTC 查询通过 Tool 执行，LLM 不直接编 SQL 或编数据。
- **最终裁决规则**：Summary 只能基于上游报告判断，客观数据和 RAG 资料优先，证据不足时输出疑似而不是强行确诊。

所以这里更准确的说法是“降低幻觉风险和不可追踪风险”，而不是声称已经量化降低了多少幻觉率。
